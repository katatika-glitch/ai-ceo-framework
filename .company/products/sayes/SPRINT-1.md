# SAYES Sprint 1 計画書

期間：2026-05-03 〜 2026-05-17
目標：Googleログイン・属性登録・AI提案エンジン・YES/NO精緻化の基幹フローを動く状態にする

---

## 技術スタック（確定）

| レイヤー | 選定技術 | 理由 |
|----------|----------|------|
| フロントエンド | Next.js 14（App Router）+ Tailwind CSS | Vercelとの親和性が高く、スマホファーストのレスポンシブUIを最小コードで実現。一人開発でもページ・API Routesを同一リポジトリで管理できる |
| ホスティング（SAYES本体） | Vercel | Next.jsとのネイティブ統合でCI/CDがゼロ設定。無料枠で十分運用可能。日本向けエッジロケーションあり |
| 認証 | NextAuth.js（Google OAuth Provider）| Next.jsとのプラグイン統合で実装工数が最小。セッション管理・JWTが自動処理される。Googleログイン必須要件と完全一致 |
| バックエンド（API） | Next.js API Routes（サーバーレス） | 追加サーバー不要。プロジェクト保存・ユーザー管理・Stripe Webhookをサーバーレス関数として一元管理 |
| データベース | Supabase（PostgreSQL）| 既存インフラ（app.pyで使用実績あり）で運用ノウハウあり。Row Level Securityでユーザーデータを安全に分離。無料枠で初期フェーズをカバー |
| AI統合 | Anthropic Claude API（claude-sonnet-4-6）| 既存モデルと統一。YES/NOコンテキスト管理・HTML生成の品質が最も安定。会話履歴はSupabaseに保存しコスト最適化 |
| 生成HTML ホスティング | Netlify API（自動デプロイ） | 要件確定済み。生成HTMLをAPIでデプロイしURLを発行する |
| 決済 | Stripe | 実績豊富・日本円対応・Webhook統合が容易。サブスク管理（Stripe Billing）も標準機能で実現 |

---

## スプリントゴール

Sprint 1終了時点で、以下の状態を達成する：

**「Googleでログインし、属性を登録した後、AIがYES/NO会話でプロダクト提案を絞り込み、1ファイルHTMLを生成してNetlifyにデプロイし、URLをユーザーに返す」フローがスマホブラウザで動作すること**

## 技術的リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| Claude APIのHTML生成品質ばらつき | 高 | プロンプトにHTML構造テンプレートを固定し出力スキーマを強制する。生成後にバリデーション処理を追加 |
| Netlify Deploy APIのレートリミット | 中 | 1ユーザーあたりのデプロイ頻度を制限（1セッション1デプロイ）。エラー時はリトライUIを表示 |
| Google OAuth同意画面の審査遅延 | 中 | 開発初期はテストユーザー登録で対応。本番公開前にOAuth審査を早めに申請 |
| Supabase RLSの設定ミスによるデータ漏洩 | 高 | RLSポリシーはテーブル作成時に必ず有効化。統合テスト時に別ユーザーのデータが見えないことを確認 |
| AI提案の重複排除ロジックの複雑化 | 低 | Sprint 1では「過去プロジェクトのtitleをプロンプトに除外リストとして渡す」シンプル実装で対応 |
| スマホブラウザのGoogle認証ポップアップブロック | 中 | NextAuth.jsのリダイレクト方式（`redirect: true`）を使用しポップアップ依存を避ける |
